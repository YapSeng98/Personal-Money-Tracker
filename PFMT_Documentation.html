📋 Overview

PFMT is a personal finance management single-page application that uses ServiceNow as a cloud database backend. Users track income and expenses, set monthly budgets, create savings goals, and view spending analytics — all synced in real time to a ServiceNow PDI via custom Scripted REST APIs.

SN Tables
7
custom tables, prefix x_887486_0_
Business Rules

auto-balance, budget spend, validation

7 days
auto-renews with saved credentials

🏗️ Architecture

GitHub Pages (index.html — Single Page App) │ │ On page load: ① ── loadState() from localStorage[pfmt_state_v2] ② ── if saved token → GET /profile to validate └─ if expired → _tryAutoLogin() with saved credentials ③ ── if valid → snLoadAll() pulls fresh data from SN │ │ On user action (add/edit/delete): ④ ── update local state + saveState() [optimistic] ⑤ ── fire-and-forget POST/PUT/DELETE to SN API │ │ ServiceNow PDI (Scripted REST APIs) ├─ /auth/{action} → login, register, logout ├─ /accounts → CRUD ├─ /transactions → CRUD + filters ├─ /budgets → CRUD + auto period ├─ /goals → CRUD + progress % └─ /profile → GET stats, PUT preferences

The app is offline-first: all data is always readable from localStorage. ServiceNow is the source of truth and is synced asynchronously in the background.

🗄️ Data Model

All tables use the scope prefix x_887486_0_. Each table is accessed via GlideRecord in the server-side scripts.

user_profile

One record per registered user. Stores credentials, preferences, and profile display data.

Field NameTypeRequiredNotes
usernameString✅Unique; stored lowercase
password_hashString✅SHA-256 hex of plaintext password
display_nameString✅Shown in UI and welcome toast
emailStringOptional; shown on profile page
currency_preferenceStringDefault: SGD
language_preferenceStringDefault: en; also supports zh
avatar_colorStringHex colour for avatar initials. Default: #8B5CF6
monthly_income_targetDecimalUsed to compute savings rate on dashboard
last_loginDateTimeBumped automatically on every valid token validation
sys_created_onDateTimeSystem field; shown as "Member since" on profile page

session

One record per active login session. Tokens are invalidated by setting expires_at to now on logout, or by the weekly prune job.

Field NameTypeRequiredNotes
user_profileReference✅Links to x_887486_0_user_profile
tokenString(40)✅32-char random hex — must fit in a String(40) field
expires_atDateTime✅Set to 7 days from creation. Logout sets this to NOW.
device_hintStringFirst 100 chars of User-Agent header at login time
⚠️
The token field must be at least String(40). The generated token is 32 chars. If the field is shorter, the token gets truncated on INSERT and every subsequent validateToken() query returns null → 401 on all API calls.

account

Bank or wallet accounts owned by a user. Balances are updated automatically by BR_UpdateAccountBalance when a transaction is confirmed.

Field NameTypeNotes

🔌 REST API Reference

Base URL

https://<your-instance>.service-now.com/api/x_887486_0/pfmt/v1

Required Headers (all endpoints except /auth/*)

HeaderValue
X-PFMT-Token32-char hex token from login response
X-HTTP-MethodGET | POST | PUT | DELETE — method override for CORS
Content-Typeapplication/json
ℹ️
CORS note: All resources must have "Requires authentication" = false in SN. Authentication is handled entirely through the X-PFMT-Token header, not SN's native auth. The X-HTTP-Method header is used so that all requests can be sent as POST to avoid browser CORS preflight issues on DELETE.

🔐 Authentication & Sessions

Login Flow

1 User enters instance + username + password 2 App POSTs to POST /auth/login 3 SN: PFMTAuthHelper.hashPassword(password) → compare against stored password_hash in user_profile 4 SN: createSession() inserts record in x_887486_0_session → token = gs.generateGUID().replace(/-/g, '') (32 hex chars) → expires_at = NOW + 7 days 5 Response: 200 with { token, user_profile_sys_id, username, ... } 6 App stores state.snToken = token → saved in localStorage 7 All subsequent requests include X-PFMT-Token: <token> 8 SN: validateToken() queries session table → WHERE token = ? AND expires_at > NOW → returns user_profile sys_id (or null → 401)

Auto-connect on Page Load

Page loads ├─ loadState() from localStorage │ ├─ [Has saved token] GET /profile to validate │ ├─ Valid → snLoadAll(), render app │ └─ Expired → _tryAutoLogin() │ └─ [No token] _tryAutoLogin() ├─ Has saved credentials → silent POST /auth/login │ ├─ OK → snLoadAll(), render app (no login screen shown) │ └─ Fail → show login screen with error └─ No credentials → show login screen (pre-filled instance)

PFMTAuthHelper — Method Reference

MethodReturnsDescription
validateToken(token)String|nullQueries session table. Returns user_profile sys_id if valid and not expired, otherwise null.
hashPassword(plaintext)StringSHA-256 hex via GlideDigest. Called on login (compare) and register (store).
generateToken()StringOne GUID with hyphens removed → 32 hex chars. Fits in String(40) field.
createSession(sysId, hint)StringInserts session record with 7-day expiry. Returns the token.
deleteSession(token)voidSets expires_at = NOW (more reliable than deleteRecord which ACLs may block).
pruneExpiredSessions()voidHard-deletes all records where expires_at < NOW. Run as a weekly scheduled job.
errorResponse(res,status,msg)voidStandardised error: sets HTTP status and body { error: "msg" }.

Security Notes

⚠️
Password hashing is SHA-256 without salt. This is adequate for a personal-use PDI but would need bcrypt/PBKDF2 + salt for a production multi-tenant system.
⚠️
Credentials stored in localStorage. The auto-connect feature saves username and password to the browser's localStorage for seamless reconnection. Do not use the auto-connect feature on shared or public computers.
ℹ️
PIN lock is a separate app-level guard. The 4-digit PIN hash is stored under a separate localStorage key. It prevents casual access to the app but does not protect the saved SN credentials — these are still in localStorage.

Data Sync Model

PFMT uses an optimistic local-first sync pattern:

  1. Every CRUD action updates local state immediately and calls saveState() to persist to localStorage.
  2. The UI re-renders from local state instantly — no waiting for the server.
  3. If state.snToken is set, an async SN API call fires in the background (fire-and-forget).
  4. After a successful POST, the returned sys_id replaces the local numeric ID so future PUT/DELETE use the SN identifier.
  5. Sync errors are shown as non-blocking warning toasts; local data is never rolled back.
  6. On page load, snLoadAll() fetches fresh data from all 4 endpoints and overwrites local state.
ℹ️
ID strategy: New items get a local integer ID (state.nextId++). After SN POST responds with a sys_id, the item's id is updated to the sys_id string. _snId(item) returns item.sys_id || (item.id if it's a 32-char string) for safe PUT/DELETE targeting.

PIN Lock Feature

  • Set a 4-digit PIN in Settings → App Security.
  • The SHA-256 hash of the PIN is stored in localStorage under a separate key (PIN_KEY).
  • The Lock sidebar item hides the app behind a numpad overlay immediately.
  • Forgot PIN: Clears all localStorage (including the PIN hash and app state) and reloads — data is re-fetched from SN on next login.

⚡ Business Rules

BR_UpdateAccountBalance

Table: x_887486_0_transaction  |  When: After · Insert + Update  |  Condition: state = '2' (Confirmed)

Keeps current_balance on the linked account in sync. On update, it reverses the previous transaction first (debits/credits the old amount back) then applies the new one. This ensures balance is always correct even if amount or type changes.

// Pseudo-logic:
if (isUpdate && previous.state === '2') {
  balance += (prevType === 'income') ? -prevAmt : +prevAmt;  // reverse
}
balance += (txnType === 'income') ? +amount : -amount;        // apply
accountGR.current_balance = balance;

BR_UpdateBudgetSpent

Table: x_887486_0_transaction  |  When: After · Insert  |  Condition: transaction_type = 'expense'

Finds the active budget for the transaction's category (where period_start ≤ today ≤ period_end). Increments spent_amount, recalculates remaining_amount. Fires the event x_pfmt.budget.threshold_reached when spending % ≥ alert_threshold.

BR_ValidateTransaction

Table: x_887486_0_transaction  |  When: Before · Insert + Update

  • Aborts if amount ≤ 0 or linked account is inactive
  • Forces future-dated transactions to Draft state (state = '1') with an info message
  • On insert of a recurring template: sets next_run_date based on recurring_frequency

BR_BudgetCalculatedFields

Table: x_887486_0_budget  |  When: Before · Insert + Update

  • Calculates remaining_amount = budget_amount − spent_amount
  • If period_start is empty, auto-sets period dates based on period_type (monthly / weekly / yearly)

🔄 Scheduled Flows

FLOW_MonthlyBudgetReset

Schedule: 1st of every month, 00:01 SGT

Iterates all budgets where period_type = 'monthly':

  1. Logs the previous period's spent amount to gs.info
  2. If rollover_enabled = true: adds unspent balance to next month's budget amount
  3. Resets spent_amount = 0 and recalculates remaining_amount
  4. Updates period_start and period_end to the new month

Outputs: budgets_reset, rollovers_applied, new_period

FLOW_RecurringTransactions

Schedule: Daily, 08:00 SGT

Finds all confirmed transactions where is_recurring = true and next_run_date ≤ today:

  1. Clones each as a new confirmed transaction (one-off instance — is_recurring = false)
  2. Sets notes to reference the template sys_id for traceability
  3. Advances the template's next_run_date by the frequency (daily +1d, weekly +7d, monthly +1m)

Output: transactions_created

🚀 Deployment Guide

  1. Create ServiceNow Tables

    In SN Studio, create 7 tables with prefix x_887486_0_: user_profile, session, account, transaction, category, budget, savings_goal. Add all fields from the Data Model section. Ensure session.token is at least String(40).

  2. Create Script Include — PFMTAuthHelper

    Studio → Create File → Script Include. Name: PFMTAuthHelper. Client callable: No. Paste SI_PFMTAuthHelper.js.

  3. Create Script Include — PFMTBudgetHelper

    Name: PFMTBudgetHelper. Client callable: Yes (required for GlideAjax). Paste SI_PFMTBudgetHelper.js.

  4. Create Scripted REST APIs

    Studio → Create File → Scripted REST API. API Name: PFMT API, Base path: pfmt/v1. Add 6 resources, setting "Requires authentication" = false on each:

    Resource PathMethodsFile
    /auth/{action}POSTREST_AuthAPI.js
    /accountsGET, POST, PUT, DELETEREST_AccountsAPI.js
    /transactionsGET, POST, PUT, DELETEREST_TransactionsAPI.js
    /budgetsGET, POST, PUT, DELETEREST_BudgetsAPI.js
    /goalsGET, POST, PUT, DELETEREST_GoalsAPI.js
    /profileGET, PUTREST_ProfileAPI.js

  5. Create Business Rules

    Studio → Create File → Business Rule. Create 4 rules as described in the Business Rules section. Ensure correct table, timing (Before/After), and operations (Insert/Update).

  6. Create Client Scripts

    Studio → Create File → Client Script. Create 3 scripts (CS_DefaultTransactionDate, CS_FilterCategoriesByType, CS_RecurringToggleAndBudgetHint) on the transaction table.

  7. Create Scheduled Flows

    Flow Designer → New → Scheduled Flow. Create FLOW_MonthlyBudgetReset (1st of month, 00:01 SGT) and FLOW_RecurringTransactions (daily, 08:00 SGT). Add a Script step in each and paste the corresponding flow JS file content.

  8. Seed Categories

    SN Navigator → Scripts – Background. Run:

    var util = new GR_Utilities();
util.seedCategories();

  9. Deploy Frontend to GitHub Pages

    Push index.html to the main branch root. Enable GitHub Pages: Repo → Settings → Pages → Source: main / root. App is live at https://<username>.github.io/<repo>/.

  10. First Login

    Open the GitHub Pages URL. Enter your SN instance URL (e.g. dev275144.service-now.com), username, and password. Click Sign In. Credentials are saved — future visits connect automatically.

❌ Error Reference

HTTP StatusMeaningCommon Cause
200Success—
201CreatedRecord inserted successfully
400Bad RequestMissing required field (e.g. amount, description, sys_id)
401UnauthorisedNo token, invalid token, or expired token. Most common if session.token field is too short (truncated token never matches).
403ForbiddenToken is valid but the requested record belongs to a different user
404Not FoundRecord sys_id not found, or unknown /auth/{action} path
405Method Not AllowedX-HTTP-Method header missing or has an unsupported value
409ConflictDuplicate budget for same category, or duplicate account name
500Server ErrorGlideRecord insert failed (check SN system logs)

All error responses use this format:

{ "error": "Human-readable message describing what went wrong" }
